# wsl-vpnroute — mirror Windows-host VPN routes into WSL2

Gives a WSL2 instance access to networks that are only reachable through **VPNs
running on the Windows host**. WSL2 cannot natively use the Windows host's VPN
routes; this bridges them: a [gvisor-tap-vsock](https://github.com/containers/gvisor-tap-vsock)
(`gvproxy`) TAP tunnel to Windows, plus a small Go daemon that watches the
Windows VPN adapters and copies their routes into WSL while a VPN is connected.

It builds on the [wsl-vpnkit](https://github.com/sakai135/wsl-vpnkit) approach
(gvproxy under WSL2) and adds the missing piece: **automatic VPN route
synchronisation** — so traffic to the VPN's subnets transparently exits through
the Windows VPN, and only while that VPN is up.

## What runs

| Piece | Where (installed) | Role |
|---|---|---|
| `wsl-gvproxy.exe` | `/usr/local/lib/vpnroute/` (runs on the **Windows** side via `/init`) | gVisor userspace network proxy — the Windows end of the tunnel |
| `wsl-vm` | `/usr/local/lib/vpnroute/` | Linux end of the tunnel; attaches gvproxy to the `wsltap` device |
| `vpnroute-monitor` | `/usr/local/sbin/` (Go, source in `monitor/main.go`) | sets up `wsltap`, starts the tunnel, and **monitors VPN adapters → syncs routes** |
| `vpnroute-discover` | `/usr/local/sbin/` (bash) | one-shot: finds Windows TAP-Windows adapters, writes `vpn-adapters.conf` |
| `wsl-vpnroute.service` | `/etc/systemd/system/` | runs `vpnroute-monitor` (`Restart=on-failure`) |
| `vpn-adapters.conf` | `/etc/vpnroute/` | GUID → friendly-name map of the Windows VPN adapters to watch |

`wsl-gvproxy.exe` and `wsl-vm` are **not part of this repository** — they are
downloaded from an upstream release at install time (see
[Upstream binaries](#upstream-binaries-attribution)).

## How it works

1. `vpnroute-monitor` creates the **`wsltap`** TAP device — `192.168.127.2/24`, gateway
   `192.168.127.1` (the standard gvproxy network), fixed MAC `5a:94:ef:e4:0c:ee` —
   and starts `wsl-vm` + `wsl-gvproxy.exe` so traffic on `wsltap` exits through the
   Windows network stack.
2. It loads `/etc/vpnroute/vpn-adapters.conf` (lines: `<lowercase-guid><two spaces><name>`)
   and maps each WSL interface to a Windows adapter GUID via `/sys/class/net/<if>/...`.
3. While a watched VPN adapter is **up**, every route on that adapter (except host/32,
   multicast, broadcast, link-local) is **replicated into WSL pointing at `192.168.127.1`
   dev `wsltap`** (priority 5), and any shadowing route on another interface is removed.
   So WSL traffic to the VPN's subnets goes through the tunnel → Windows → the VPN.
4. A **3 s poll** detects VPN disconnect (WSL2 mirrored networking doesn't reliably emit
   `RTM_DELROUTE`); on disconnect the daemon removes the routes it added.

`wsltap` is **not** the default route — your normal path stays on its existing
interface. `wsl-vpnroute` only adds routes for the VPN-reachable subnets, and only
while the corresponding Windows VPN is connected.

## Config: `/etc/vpnroute/vpn-adapters.conf`

```
00000000-0000-0000-0000-000000000000  OpenVPN Office
11111111-1111-1111-1111-111111111111  OpenVPN Home
```
(GUIDs and names are examples — yours are filled in automatically by `vpnroute-discover`.)
Generated by `vpnroute-discover`, which runs Windows PowerShell
(`Get-NetAdapter | where InterfaceDescription -like 'TAP-Windows*'`) and writes the
GUID+name of each TAP-Windows adapter. `install.sh` runs it for you on install.
If no adapters are found it writes an **empty** config (not an error) and the service
idles. **Re-run `vpnroute-discover` whenever you add, remove, or recreate a Windows VPN
adapter** (it restarts the service if the file changed).

## Install / operate

Requirements: a WSL2 distro with `systemd`, plus `go`, `curl`, `sha256sum`, and
`jq` available (`jq` is used by `vpnroute-discover`).

```bash
# Install (from the repo root): downloads + checksum-verifies the upstream binaries,
# builds the monitor, installs the service, runs discovery, and starts it.
sudo ./install.sh

# Operate:
systemctl status wsl-vpnroute
journalctl -u wsl-vpnroute -f  # logs "+ <subnet> via 192.168.127.1 ..." as VPNs come/go
sudo systemctl restart wsl-vpnroute
ip route show dev wsltap       # subnets currently tunnelled (empty = no watched VPN up)
```

This is developer-facing tooling: the installer **fails loudly** on any error
(download failure, checksum mismatch, missing dependency, build failure) and
never silently falls back.

## Upstream binaries (attribution)

`wsl-vpnroute` does not ship the userspace network binaries. `install.sh`
downloads them from a **pinned** upstream release and verifies their SHA-256
checksums before installing:

| Installed as | Source | Upstream asset | Release |
|---|---|---|---|
| `wsl-gvproxy.exe` | [containers/gvisor-tap-vsock](https://github.com/containers/gvisor-tap-vsock) (Apache-2.0) | `gvproxy-windows.exe` | `v0.8.9` |
| `wsl-vm` | [containers/gvisor-tap-vsock](https://github.com/containers/gvisor-tap-vsock) (Apache-2.0) | `gvforwarder` | `v0.8.9` |

The `wsl-gvproxy.exe` / `wsl-vm` naming convention follows
[sakai135/wsl-vpnkit](https://github.com/sakai135/wsl-vpnkit) (MIT). To bump the
upstream version, change `GV_VERSION` and the pinned `*_SHA256` values in
`install.sh` (checksums come from the release's `sha256sums` file).
